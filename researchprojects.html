Software Defined Radio for Real-time Wireless Communication Systems
Funded by National Science Foundation

From the Internet of Things (IoT) used in a smart home to large-scale space shuttles, secured and robust wireless communication system in one of the most rudimentary requirements. Software Defined Radio (SDR) marks the transition between traditional hardware radios and digitally reprogrammable and robust wireless devices. In this platform, all the physical layer operations and some network layer operations of a network model are defined using software. With a high-speed processor, an antenna and a receiver being the only hardware peripherals of the system, software defined radio provides the desired flexibility, configurability, and programmability in wireless communications by providing the functionality to change digital operations and modulation schemes on-the-fly.

I worked under the supervision of Dr. Song Han and Dr. Shengli Zhou on using software defined radio to implement real-time wireless communication systems. During my research, I explored GNURadio software framework for software defined radio platform and used the framework to implement an ad hoc wireless network capable of inter-computer message transfer. I used a lot of C++ (Boost/SWIG), and a little bit of Python to code the signal processing blocks in GNURadio. I also experimented with IEEE 802.15.4 ZigBee implementation in GNURadio to better understand wireless data transfer in the software defined radio platform.


GNURadio

Compared to the other software frameworks available for software defined radio such as MATLAB or LabVIEW Communications, GNURadio stands out by providing transparency, the ability to demystify functions of the signal processing blocks in the library with the help of open-source codes, and the ability to create custom signal processing blocks that can perform any desired functionality. For my project on creating ad-hoc network, I used GNURadio signal processing blocks for running the PHY layer, and a mac layer was written on top of it using Python for establishing socket connections and controlling/triggering message transmissions. GNURadio’s packet encoder block was used to packetize the data in the physical layer. Along with creating a wireless ad hoc network, the project helped us understand how GNURadio performs signal processing, digital data transmissions, and packet transmission.

As GNURadio is open-source, the codes for all the signal processing blocks are available on Github. Upon inspecting the codes related to the Packet Encoder block, we found that the block wraps the data into a packet of a given payload length with a preamble, access code, a header, and a tail. The final payload in the packet contains a checksum, typically a CRC value, which is used for validation of the packet contents. An 8 bit tail is added at the end for synchronization purpos.

In order to validate the packet model, a test was set up incorporating GNURadio's Socket PDU blocks. In this system, messages can be written and read from socket ports for controlled transmissions. Including Socket PDUs would also allow a mac layer written on top of it to connect to the socket ports.

In the test above, specified sample rate is 250*103, with a samples per symbol of 4.
So, symbol rate = (250*103)/4 = 62.5*103
For GMSK modulation, 1 bit is used per symbol.
So, bit rate = 1 * 62.5*103 = 62.5 kb/s
The specified payload length in the Packet Encoder block is 10 bytes or, (10 * 8) bits = 80 bits.
Using our estimation of the packet structure, The total number of packet bits = (Preamble bits) + (Access bits) + (Header bits) + (Payload bits) + (CRC bits) + (Tail bits) = 248 bits
As a result, packet duration = 248/62.5 ms ≈ 4 ms

In order to test the estimation, I ran the physical layer in GNURadio and wrote a Python script that connects to the socket and continuously sends the message “hello world” every 40 ms. Ad-hoc Network

The ad-hoc network I worked on used the Socket PDU blocks in the physical layer. On top of the GNURadio physical layer, a mac layer was written using Python. The mac layer connects to the sockets using Python’s socket library functions. In order to cover different scenarios such as multiple sender or multiple receiver, the mac layer needed to add a header with the required information on it before sending out the message. In the sender mac layer, sender sends the message with a header that includes the information of the sender’s name, the designated receiver’s name, and the message length. The receiver mac layer then receives the entire message and retrieves all the information needed from the header. It then sends out an acknowledgments of that includes the information of the receiver’s name and how many characters were successfully received.

A monitor was also used that only had a listener which listened to every message being transmitted without checking the designated receiver the message is being sent to. There is also a broadcast functionality for the sender. If broadcast is chosen, the sender name in the mac header is replaced with “broadcast”. When the receiver sees that the message is a broadcast, it goes ahead and receives the message. During a broadcast, no acknowledgment is sent back from the receiver side. IEEE 802.15.4 ZigBee on GNURadio

We studied the IEEE 802.15.4 network implementatio from the WIME project. IEEE 802.15.4 is a widely used standard for low data-rate wireless personal area networks (WPAN). This standard defines the physical layer and media access control (MAC) layer of the Open Systems Interconnection (OSI) network model. The entire thing needed to be set up in a linux environment. In order to monitor traffic, Wireshark was used. The system included an O-QPSK PHY layer, with custom written MAC and Contiki 2.7 RIME network stack as the APP layer.

I collaborated with Dr. John Ayers to extend his research on studying buffer layers for metamorphic semiconductor devices. In my work, I simulated and analyzed novel growth platforms for InGaAs/GaAs semiconductor devices. I implemented algorithms on MATLAB to simulate chirped and unchirped superlattice structures, and quantify their performance using theories from Numerical Energy Minimizatin Model, Electric Circuit Model for Strained Layer Epitaxy, and Matthews, Mader, and Light approach.

Metamorphic realization of mismatched semiconductor devices such as high electron mobility transistors (HEMTs), light-emitting diodes (LEDs), and multi-junction solar cells is of great interest due to the flexibility it affords in designing layer compositions and thicknesses. A drawback of metamorphic growth is the inherent incorporation of misfit dislocations and their associated threading defects, which can degrade device performance. It is possible to minimize these effects by using a suitable buffer layer grown between the device and its mismatched substrate, and the linearly-graded buffer has been the moststudied approach. Use of a grading coefficient less than 500 cm-1 has been found to greatly reduce the threading dislocation density at the top of such a linearly-graded buffer. On the other hand, superlattices have been used either alone or in conjunction with graded layers to filter threading dislocations.

Strain and threading dislocations affect metamorphic device performance. Metamorphic devices often use graded buffers to accommodate the lattice mismatch. However, it has also been shown that strained layer superlattice (SLS) buffers have the ability to filter dislocations.

Throughout the research we evaluated the average and surface strain as well as the surface in-plane lattice constant for four types of superlattices using an electric circuit model for semiconductor strained epitaxy [1]. Type I is compositionally chirped and type II is thickness chirped while types III and IV are unchirped superlattices with two different average compositions, as shown in the figure below.

From our study, we saw that variations of the top composition allows a wider variation of indium equivalent than variation of the total thickness. Superlattices of type I, III, or IV provide a wide variation of indium equivalent from approximately 2% to 45%, whereas type II superlattices provide a much narrower range of surface lattice constant

We also modeled chirped and unchirped strained-layer superlattices (SLSs) as dislocation filters for metamorphic semiconductor devices. Chirped SLSs combine the mismatch accommodating properties of graded buffers with the dislocation filtering properties of SLSs. We compared the dislocation filtering behavior of chirped superlattices (both compositionally-modulated an thickness-modulated) to unchirped superlattices and linearly-graded layers. Type II chirped superlattices can outperform unchirped superlattices and linearly-graded layers as dislocation filters.
